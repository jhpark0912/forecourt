# 설계: Forecourt — 부동산 앞마당(관심단지 시세 인사이트 도구)

작성: office-hours 세션 / 2026-06-24
모드: Builder (개인용 도구 — "내 문제를 긁는" 사이드 프로젝트)
상태: DRAFT

---

## 1. 문제 정의 (Problem Statement)

내가 등록한 아파트 단지들의 **실거래가 변동과 시세 흐름을 한눈에** 보고 인사이트를 얻고 싶다.
호갱노노·아실·네이버부동산·KB가 이미 많은 걸 제공하지만, 두 가지가 불편하다:

1. **관심단지가 흩어져 있다.** 매번 검색해야 하고, 내 단지만 모아 보는 워치리스트가 없다.
2. **내가 원하는 지표가 없다.** 단지간 상대비교, 평단가 정규화 같은 나만의 계산을 못 본다.

→ 이건 차별화 마케팅이 아니라 **본인 워크플로우 최적화**가 목적인 도구다. 가장 좋은 사이드 프로젝트의 형태.

## 2. 핵심 가치 (What Makes This Useful)

관심단지를 모아 **나만의 지표로 빠르게** 본다. 핵심 지표 4종:

- **평단가 정규화** — 평형·면적이 달라도 평당 단가로 환산해 공정 비교 (모든 비교의 기본 단위)
- **단지간 상대비교** — A·B·C 단지를 평당가/변동률로 나란히, "지금 어디가 상대적으로 싼가"
- **기간별 변동추이** — 최근 3/6/12개월 실거래 추세 그래프와 변동률
- **가격 알림/이벤트** — 신고가/급변 감지 (MVP는 접속 시 하이라이트, 추후 푸시)

## 3. 확정된 전제 (Premises — 사용자 동의 완료)

1. **데이터는 '실거래가'가 기본, '호가 시세'는 보류.** 국토부 실거래가 공개 API는 무료·합법·안정적.
   호가(매물) 시세는 공식 API가 없어 스크래핑 필요 → 약관 리스크 + 잘 깨짐 → MVP 제외.
2. **알림은 MVP에서 '접속 시 하이라이트'로 시작.** 실시간 푸시는 스케줄러 인프라가 무거움 → 가치 검증 후 도입.
3. **속도의 답은 '내 DB 캐싱'.** 국토부 API는 느리고 호출 제한 있음 → 단순 프록시가 아니라
   **데이터 적재 파이프라인 + 조회 서비스** 구조. 받아온 데이터를 내 DB에서 서빙.
4. **화면의 중심은 워치리스트 3종.** 워치리스트 → 비교 테이블 → 변동추이 차트. 검색은 보조.

## 4. 핵심 데이터 설계 결정 (가장 중요)

**"현재 시세만 캐싱"으로는 변동추이·신고가 알림이 불가능하다. 이력을 쌓아야 한다.**

- **백필(최초 등록 시):** 관심단지 추가 순간, 국토부 API에서 과거 **3~5년치** 월별 실거래를
  한 번에 수집해 적재. (API가 `지역코드 + 거래년월` 단위 과거 조회 지원)
- **증분(이후):** 최신 1~2개월만 주기적으로/접속 시 재수집, 누락분만 추가.
- **저장 단위:** "현재 시세 한 줄"이 아니라 **단지별 개별 실거래 이력 테이블**
  (거래일·평형·전용면적·층·거래금액). 변동추이·평단가정규화·신고가판정 전부 이 테이블에서 파생.
- **데이터 양:** 단지당 월 수 건 수준 → 단지 수십 개 × 5년치라도 수만 행, 수십 MB. 가볍다.

### 4.1 데이터 소스 참조 (국토부 실거래가 API)
- 상세자료 엔드포인트: `https://apis.data.go.kr/1613000/RTMSDataSvcAptTradeDev/getRTMSDataSvcAptTradeDev`
- 파라미터: `serviceKey`, `LAWD_CD`(시군구 5자리), `DEAL_YMD`(YYYYMM), `pageNo`, `numOfRows`
- 조회 단위: **시군구 1곳 × 1개월** (단지 단위 조회 불가 → 응답을 단지명으로 필터링)
- 키 발급: data.go.kr 활용신청(무료). 응답에 단지 고유 ID 없음(단지명+법정동+면적으로 식별).

## 5. 검토한 접근 (Approaches)

### Approach A — 로컬 단일 앱 (최소 실행 가능)
Spring Boot + 임베디드 DB + 가벼운 프론트. 스케줄러/푸시 없음. 인프라 0. 노력 S~M / 리스크 낮음.

### Approach B — 제대로 된 데이터 파이프라인 (이상적 구조) ← 선택
Spring Boot + Postgres + 스케줄러(매일 증분) + 알림 감지. 도메인 분리. 노력 L / 리스크 중.

### Approach C — 봇 우선 / 서버리스
배치 수집 → 텔레그램 봇 리포트. UI 최소. "매일 알아서 알림"엔 강력하나 UX 한계.

## 6. 추천 구조 (Recommended Approach)

**Approach B + 토폴로지 1 (사용자 환경 맞춤)**

선택 근거: 사용자가 **유지보수성**과 **Java/Spring 환경 유지**에 일관되게 무게를 둠.
GitHub Actions 수집기(토폴로지 2)는 인프라는 더 가볍지만 익숙지 않은 스크립트 언어로 가야 해
*사용자 기준* 유지보수가 나빠짐. "유지보수성"은 절대값이 아니라 누구 기준이냐의 문제.

### 스택
- **백엔드:** Spring Boot + JPA (수집·분석·조회 전부 Java, `@Scheduled`로 매일 증분 수집)
- **DB:** **Supabase** (관리형 Postgres + 배터리). 상세 활용은 아래 §6.3.
  - **핵심: Supabase도 결국 표준 Postgres.** Spring은 JDBC로 그냥 붙고(JPA/Hibernate Postgres dialect), `pg_dump`로 언제든 RDS/Neon/docker-compose로 이전 → 락인 없음.
  - 데이터 규모(수십 MB)는 무료 티어 500MB에 한참 못 미침.
- **백엔드 호스팅:** Fly.io / Railway 무료 (idle 시 잠듦, 개인용 무해)
- **프론트:** Vercel(Next/정적) 또는 Spring 서버사이드로 통합 — UI 양에 따라 결정

### 도메인 구조 (사용자 컨벤션: domain/ vs global/ vs api/)
- `domain/collector` — 국토부 API 수집·백필·증분 (수집 로직)
- `domain/market` — 평단가 정규화·변동률·신고가 판정 (분석 로직)
- `domain/watchlist` — 관심단지 등록·관리
- `api/` — 프론트용 조회 컨트롤러 (워치리스트/비교/추이)
- `global/` — 설정·예외·API 클라이언트 공통

### 6.3 Supabase 활용방안

원칙: **Supabase는 "Postgres + 필요한 배터리만" 골라 쓴다.** 분석·매칭 등 핵심 로직은 Spring에 둔다
(유지보수 일관성 — 사용자 선호). Supabase에 비즈니스 로직을 과하게 밀어넣지 않는다.

**A. DB로서 (필수, 기본)**
- Spring Boot ↔ Supabase Postgres를 **JDBC로 연결**. 별도 SDK 불필요, 그냥 커넥션 문자열만 바꾸면 됨.
- 연결 방식 두 가지:
  - **Direct (5432):** 상주 백엔드(Spring/HikariCP)는 이걸로. 커넥션 풀을 앱이 직접 관리.
  - **Transaction Pooler (6543, pgbouncer):** 서버리스·짧은 함수에서 붙을 때. (Edge Function 등)
- 마이그레이션은 Flyway/Liquibase로 Spring에서 관리 → 스키마 버전 관리 유지.

**B. 선택적으로 켤 배터리 (가치 있을 때만)**
- **Auth** — 단일 사용자 로그인을 직접 안 만들고 Supabase Auth(이메일/소셜)로 해결. 관심단지는 '내 데이터'라
  Row Level Security(RLS)로 사용자별 격리. → 로그인·보안 코드 절약.
- **Realtime** — 실거래 적재 시 테이블 변경을 프론트가 구독 → 신고가 하이라이트를 폴링 없이 실시간 갱신.
  단 MVP는 "접속 시 계산"으로 충분 → **후순위**.
- **Database Webhooks / pg_cron** — 적재 후 신고가 감지 트리거 → 텔레그램 호출. **알림 단계에서 활용**.
- **안 씀:** Storage(저장할 파일 없음).

**C. 스케줄러 — Supabase가 빛나는 지점**
- 함정: Fly/Railway 무료 백엔드는 idle 시 잠들어 **Spring `@Scheduled`가 안 돈다.** 매일 증분 수집이 멈춤.
- 해법 두 가지:
  - (a) 외부 cron(cron-job.org)이 수집 엔드포인트를 호출해 백엔드를 깨움 — 단순.
  - (b) **Supabase pg_cron + Edge Function**이 매일 수집을 트리거 — DB 안에서 스케줄이 돌아 상주 서버 의존 제거.
- → Supabase를 쓰는 이상 (b)가 자연스러운 선택지. 수집 로직 본체는 Spring 엔드포인트에 두고, pg_cron이 그걸 호출하는 형태도 가능.

**D. 무료 티어 함정 (반드시 인지)**
- Supabase 무료 프로젝트는 **7일간 활동 없으면 자동 일시정지(pause)**. (Neon의 scale-to-zero와 다른 점 — 깨우는 데 수동/지연 발생 가능.)
- 완화: 위 (b) pg_cron이 매일 돌면 자연히 활성 유지되어 일시정지 안 걸림. 즉 스케줄러가 keep-alive 역할도 겸함.
- DB 백업도 무료 티어는 제한적 → 중요해지면 주기적 `pg_dump`를 별도로.

## 7. 화면 (UI — 3종이 80%)

1. **워치리스트** — 내 관심단지 카드/리스트. 평당가·최근 변동률·신고가 여부 하이라이트.
2. **비교 테이블** — 선택 단지들을 평당가/기간 변동률로 나란히. 정렬 가능.
3. **변동추이 차트** — 단지별 3/6/12개월 실거래 추세 라인.
- 보조: 단지 검색(워치리스트에 담기 위한 것).
- 상태 처리: 로딩 / 빈 워치리스트 / 데이터 없음(거래 희소 단지) / 수집 실패.

## 8. 관심단지 등록 — 주소→법정동코드 해석 (지도 활용)

**문제:** 국토부 실거래가 API는 `LAWD_CD`(법정동 시군구 5자리)를 입력으로 받는데, 사용자는 코드를 모른다.
사용자가 아는 건 "단지명" 또는 "주소"뿐. 그 간극을 **지도/지역검색으로 메운다.**

### 등록 흐름
```
사용자: 지도에서 단지 클릭  또는  "강남구 대치동" / 단지명 검색
   ↓
지도·지역 API: 좌표 + 주소 + 법정동코드(10자리) 반환
   ↓
앞 5자리 추출 → LAWD_CD (국토부 API 파라미터)
   ↓
관심단지 레코드에 코드·단지명·법정동 저장 → 이후 수집·필터링에 사용
```

핵심: **지도 API가 주는 법정동코드 앞 5자리 = 국토부 API의 LAWD_CD.**
별도 법정동코드 매핑 테이블을 직접 관리할 필요가 없어진다.

### 지도/주소 API 선택지

| 후보 | 강점 | 비고 |
|------|------|------|
| **카카오 지도 API** (추천) | 지도 UI + 키워드(단지명) 검색에서 좌표·법정동코드 한 번에. 무료 쿼터 넉넉(일 30만) | JS 키(지도)·REST 키(검색) 분리 |
| **VWorld** (국토부 공간정보) | 지오코더가 법정동코드 직접 반환. 실거래가와 같은 정부 데이터 생태계 | 지도 UX는 카카오보다 약함 |
| **juso.go.kr** (행안부 도로명주소) | 주소검색 시 `admCd`(법정동코드) 반환 | 지도 없음, 주소검색만 |

→ 지도 UX까지 원하면 **카카오**, 순수 주소→코드 변환만이면 **juso/VWorld**.

### 등록 시 저장 필드
```
Watchlist 단지 = {
  단지명,                 // 응답 필터 키 (필수)
  법정동코드(10자리),      // 지도 API 반환값
  LAWD_CD(5자리),         // 앞 5자리 → 국토부 API 파라미터 (필수)
  법정동(읍면동),          // 동명이 단지 구분 (권장)
  좌표(lat, lng)          // 지도 표시·근처단지 비교용 (선택, UI 안 쓰면 생략)
}
```

### 주의 — 좌표는 선택, 단지명 매칭은 필수
- **좌표(lat/lng)는 데이터 파이프라인에 불필요.** 지도에 그릴 때만 필요한 UI 전용 값.
- 진짜 중요한 건 **단지명 + 법정동 + 면적** 조합. 국토부 응답에 단지 고유 ID가 없어,
  이 셋으로 "내 단지" 거래를 골라내야 한다. 지도가 주는 정규화된 단지명이 매칭의 출발점이 된다.
- 지도 API 단지명과 국토부 응답 단지명이 표기상 다를 수 있으므로(공백·괄호·차수),
  매칭은 **법정동코드로 1차 좁히고 → 단지명 정규화 후 비교 → 면적으로 평형 구분** 순서.

## 9. 성공 기준 (Success Criteria)

- 관심단지 N개를 등록하면 3년치 실거래가 자동 백필되어 추이 그래프가 그려진다.
- 단지 2~3개를 평당가 기준으로 한 화면에서 비교할 수 있다.
- 접속 시 신고가/급변 단지가 시각적으로 구분된다.
- 조회는 내 DB에서 나오므로 체감 즉시 (국토부 API 직접 호출 대비 빠름).

## 10. 배포/유통 (Distribution)
- 개인용 웹앱. 백엔드는 Fly.io/Railway 무료, DB는 **Supabase** 무료, 프론트는 Vercel.
- CI/CD: GitHub → 호스팅 자동 배포(추후). MVP는 수동 배포로 시작 가능.

## 11. 열린 질문 (Open Questions)

- **Supabase 배터리 채택 범위** — DB는 확정(§6.3). Auth·Realtime·pg_cron을 어디까지 쓸지는 단계별로 결정.
  (MVP: DB만 → 알림 단계: pg_cron/Webhook → 필요 시 Auth.) 표준 Postgres라 나중에 Neon/RDS 이전 비용 거의 0.
- **수집기 하이브리드** — 조회는 Spring, 수집만 GitHub Actions로 뺄지는 운영해보고 판단. (지금 결정 불필요)
- **호가 시세** — 실거래가만으로 부족하다고 느껴지면 그때 별도 검토. (스크래핑 리스크 재평가)
- **국토부 API 키 발급** — data.go.kr 실거래가 API 신청 필요. (법정동코드는 §8 지도/지역검색으로 해석하므로 매핑 테이블 직접 관리 불필요.)
- **지도 API 키** — 카카오 채택 시 JS 키·REST 키 발급 필요. (§8)
- **단지 식별** — 국토부 API는 '단지명+지역+면적' 기반이라 단지 ID가 깔끔하지 않음. 단지 정규화/매칭 로직이 숨은 난이도. (백필 설계 시 먼저 검증할 것)
