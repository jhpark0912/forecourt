# 설계: Forecourt — 부동산 앞마당(관심단지 시세 인사이트 도구)

작성: office-hours 세션 / 2026-06-24
모드: Builder (개인용 도구 — "내 문제를 긁는" 사이드 프로젝트)
상태: DRAFT

---

## 1. 문제 정의 (Problem Statement)

내가 등록한 아파트 단지들의 **실거래가 변동과 시세 흐름을 한눈에** 보고 인사이트를 얻고 싶다.
호갱노노·아실·네이버부동산·KB가 이미 많은 걸 제공하지만, 두 가지가 불편하다:

1. **관심단지가 흩어져 있다.** 매번 검색해야 하고, 내 단지만 모아 보는 워치리스트가 없다.
2. **내가 원하는 지표가 없다.** 단지간 상대비교, 평단가 정규화 같은 나만의 계산을 못 본다.

→ 이건 차별화 마케팅이 아니라 **본인 워크플로우 최적화**가 목적인 도구다. 가장 좋은 사이드 프로젝트의 형태.

## 2. 핵심 가치 (What Makes This Useful)

관심단지를 모아 **나만의 지표로 빠르게** 본다. 핵심 지표 4종:

- **평단가 정규화** — 평형·면적이 달라도 평당 단가로 환산해 공정 비교 (모든 비교의 기본 단위)
- **단지간 상대비교** — A·B·C 단지를 평당가/변동률로 나란히, "지금 어디가 상대적으로 싼가"
- **기간별 변동추이** — 최근 3/6/12개월 실거래 추세 그래프와 변동률
- **가격 알림/이벤트** — 신고가/급변 감지 (MVP는 접속 시 하이라이트, 추후 푸시)

## 3. 확정된 전제 (Premises — 사용자 동의 완료)

1. **데이터는 '실거래가'가 기본, '호가 시세'는 보류.** 국토부 실거래가 공개 API는 무료·합법·안정적.
   호가(매물) 시세는 공식 API가 없어 스크래핑 필요 → 약관 리스크 + 잘 깨짐 → MVP 제외.
2. **알림은 MVP에서 '접속 시 하이라이트'로 시작.** 실시간 푸시는 스케줄러 인프라가 무거움 → 가치 검증 후 도입.
3. **속도의 답은 '내 DB 캐싱'.** 국토부 API는 느리고 호출 제한 있음 → 단순 프록시가 아니라
   **데이터 적재 파이프라인 + 조회 서비스** 구조. 받아온 데이터를 내 DB에서 서빙.
4. **화면의 중심은 워치리스트 3종.** 워치리스트 → 비교 테이블 → 변동추이 차트. 검색은 보조.

## 4. 핵심 데이터 설계 결정 (가장 중요)

**"현재 시세만 캐싱"으로는 변동추이·신고가 알림이 불가능하다. 이력을 쌓아야 한다.**

- **백필(최초 등록 시):** 관심단지 추가 순간, 국토부 API에서 과거 **3~5년치** 월별 실거래를
  한 번에 수집해 적재. (API가 `지역코드 + 거래년월` 단위 과거 조회 지원)
- **증분(이후):** 최신 1~2개월만 GitHub Actions cron이 주기 재수집(해당 월 교체 적재). 상세 §4.2.
- **저장 단위:** "현재 시세 한 줄"이 아니라 **단지별 개별 실거래 이력 테이블**
  (거래일·평형·전용면적·층·거래금액). 변동추이·평단가정규화·신고가판정 전부 이 테이블에서 파생.
- **데이터 양:** 단지당 월 수 건 수준 → 단지 수십 개 × 5년치라도 수만 행, 수십 MB. 가볍다.

### 4.1 데이터 소스 참조 (국토부 실거래가 API)
- 상세자료 엔드포인트: `https://apis.data.go.kr/1613000/RTMSDataSvcAptTradeDev/getRTMSDataSvcAptTradeDev`
- 파라미터: `serviceKey`, `LAWD_CD`(시군구 5자리), `DEAL_YMD`(YYYYMM), `pageNo`, `numOfRows`
- 조회 단위: **시군구 1곳 × 1개월** (단지 단위 조회 불가 → 응답을 `aptSeq`/단지명으로 필터링)
- 키 발급: data.go.kr 활용신청(무료). REST API는 자동승인. **요청 시 User-Agent 헤더 필수**(없으면 게이트웨이가 `Request Blocked` 400으로 차단). Encoding 인증키를 URL에 그대로 사용.
- **응답에 단지 식별자 `aptSeq` 존재(검증 완료, 2026-06-25).** 형식 `시군구코드-단지일련번호`(예: `11680-364`). 같은 단지=같은 seq로 월·거래 무관 고정 → 단지 매칭의 1차 키. (단지명 퍼지매칭은 보조로 강등. §8 참조)
- 주요 필드: `aptSeq`·`aptNm`(단지명)·`excluUseAr`(전용면적㎡, **공급면적 없음**)·`dealAmount`(만원, `"550,000"` 콤마 문자열)·`floor`·`dealYear/Month/Day`·`umdNm`(법정동)·`jibun`·`buildYear`. 응답은 **XML**.
- 페이징 함정: `numOfRows` 기본 10건 → 크게(1000) 주고 `totalCount`로 페이징(강남구 1개월 ≈ 431건).
- 호출 제한: 상세 엔드포인트는 **하루 1,000건** 제한(검증, 2026-06-26). 구 1개×5년 ≈ 60~120 호출이라 개인 규모는 여유.

### 4.2 적재·수집 운영 모델 (2026-06-26 확정 — 3자 대면 검토)

**수집/적재 단위 = 시군구(`LAWD_CD` 5자리) × 월(`deal_ym`).** API가 시군구 단위로만 줘서 강제됨.
- 지리 단위는 **"구(시군구)"가 유일.** 동(읍면동)은 못 받음 → 저장 후 표시·필터(`umdNm`)일 뿐. 시 전체는 그 시의 구들을 루프.
- 저장은 **받아온 구 전체를 그대로** — 관심단지로 거르지 않음(어차피 구로 받고 데이터 싸다 ≪ 무료 500MB, 이웃 비교도 공짜). 조회단(뷰/프론트)이 `aptSeq`로 필터.

**적재 = (`LAWD_CD`, `deal_ym`) 단위 "교체 적재"(멱등).** row 단위 upsert 아님.
- 국토부엔 거래 고유 ID가 없고 **완전중복 행이 실재**(같은 단지·면적·층·금액·일자 2건)라 자연키 유일성 보장 불가.
- → 한 (구, 월)을 **단일 트랜잭션으로 통째 교체**: `BEGIN → DELETE(lawd_cd, deal_ym) → bulk INSERT → COMMIT`. 재수집 멱등, 거래 정정·해제 자동 반영, 진짜 중복도 원본 보존.
- PK = surrogate `id`(bigserial), 자연 unique 제약 없음. **중복 제거 로직 없음**(완전중복은 평단가 평균 불변 → 금도금).
- **`cdealType`(해제여부) 매핑 필수** — 취소 거래가 가짜 신고가를 만들지 않게 지표에서 제외(`WHERE cdeal_type IS NULL`). (+선택 `dealingGbn` 거래유형)

**`collection_job` — 수집 상태 추적(비동기 파이프라인 필수).**
- `collection_job(lawd_cd, deal_ym_range, status[queued/running/done/failed], updated_at)`.
- ① 프론트가 "수집 중 → 완료/실패"를 **폴링**으로 앎(Realtime 자동 갱신은 금도금) ② **멱등 가드** — 같은 구가 이미 `running`이면 새 트리거 무시(중복 디스패치 방지).

**수집 2종 (성격이 다름).**
- **백필** — 등록/'받기 승인' 시 1회, 그 구의 과거 3~5년을 월별 루프.
- **증분** — 매일 Actions cron, 최근 1~2개월만 교체 재수집. **데일리 cron은 "첫 데이터"의 출처가 아니다**(신선도 유지용). 트리거 상세 §6.3·§7.

## 5. 검토한 접근 (Approaches)

### Approach A — 로컬 단일 앱 (최소 실행 가능)
Spring Boot + 임베디드 DB + 가벼운 프론트. 스케줄러/푸시 없음. 인프라 0. 노력 S~M / 리스크 낮음.

### Approach B — 제대로 된 데이터 파이프라인 (이상적 구조) ← 선택
Spring Boot + Postgres + 스케줄러(매일 증분) + 알림 감지. 도메인 분리. 노력 L / 리스크 중.

### Approach C — 봇 우선 / 서버리스
배치 수집 → 텔레그램 봇 리포트. UI 최소. "매일 알아서 알림"엔 강력하나 UX 한계.

## 6. 추천 구조 (Recommended Approach)

**서버리스 토폴로지 (3자 대면 검토로 확정, 2026-06-25)**

> 초기엔 "Spring on Fly/Railway(토폴로지 1)"였으나, 검토 결과 **상주 백엔드 서버를 두지 않는 구조**로 전환.
> 결정적 통찰: 이 도구는 **수집기(배치 job)** 와 **뷰어(화면)** 둘로 분리되며, 둘 다 *상주 서버가 필요 없다.*
> 수집기는 스케줄 job(GitHub Actions)이고, 뷰어는 정적 프론트가 Supabase를 직접 읽는다(PostgREST).
> Fly/Railway는 2024년 무료 티어 폐지로 더는 무료도 아님 → 상주 서버 회피가 비용·운영 양면에서 정답.

### 스택 (서버 0개)
```
[GitHub Actions cron]──매일──▶ Java 수집기(JAR)         [정적 프론트] 📱폰
  (무료 스케줄러)               · 국토부 API(UA헤더)        Vercel/Cloudflare Pages
                                · aptSeq 매칭·평단가 정규화   (무료·CDN·항상 켜짐)
                                · 지표 미리 계산                   │ 직접 읽기/쓰기(HTTPS)
                                      │ upsert                     │ + Google 로그인
                                      ▼                            ▼
                              [Supabase] 거래이력·지표뷰·Auth·RLS·Storage
                                ▲ 백필(등록/'받기 승인'→자동 트리거, 폴백: 로컬/수동)
```
- **수집기(Java):** 국토부 수집·`aptSeq` 매칭·평단가 정규화·지표 계산. **상주 아님 — 배치 JAR.** 무거운 로직은 전부 여기(=Java가 빛나는 곳).
  - 증분: **GitHub Actions cron**(무료, 언어무관, idle-sleep 문제 원천 제거).
  - 백필(3~5년, 등록/'받기 승인' 시 1회): **목표는 자동 트리거** — 등록 또는 '받기 승인'(쓰기) → Supabase DB 웹훅 → Edge Function → GitHub `repository_dispatch` → Actions가 그 구 백필(§9 "자동 백필" 약속과 일치). 폴백/부트스트랩은 로컬 JAR 또는 Actions `workflow_dispatch`. 과거는 안 변하므로 재실행 없음. 상세 §4.2·§6.3.
- **DB+읽기API:** **Supabase** Postgres + **PostgREST 자동 API** + RLS + Auth + Storage. 상세 §6.3.
  - 표준 Postgres라 `pg_dump`로 언제든 이전 → 락인 없음. 데이터 수십 MB ≪ 무료 500MB.
- **뷰어(프론트):** 정적 SPA가 **Supabase를 직접 읽기/쓰기**(서버 경유 없음). Vercel/Cloudflare Pages 무료. 콜드스타트·슬립 없음 → 폰에서 즉시.
- **읽기/쓰기 분리(중요):** 조회(프론트→PostgREST)는 **DB 읽기 전용 — 국토부를 호출하지 않는다.** 데이터 유입은 오직 수집(백필/증분). 미수집 지역 조회는 자동 페치가 아니라 **'받기 승인' 온램프**로 처리(§7).
- **상주 백엔드 서버: 없음.** (나중에 Java 쿼리 API가 정말 필요해지면 그때 Render/Cloud Run에 얹음 — 지금 불필요.)

### 도메인 구조
- **Java 수집기 프로젝트** (`domain/` vs `global/`):
  - `domain/collector` — 국토부 API 수집·백필·증분
  - `domain/market` — 평단가 정규화·변동률·신고가 판정 → 지표를 Supabase에 미리 계산·적재
  - `global/` — 설정·예외·국토부 API 클라이언트 공통
  - ※ 기존 `domain/watchlist`·`api/`는 Java에서 빠짐 → 워치리스트·조회는 Supabase 테이블+RLS+프론트가 담당.
- **정적 프론트 프로젝트** (별도): 로그인·워치리스트·비교·차트·메모·사진 UI.

### 6.3 Supabase 활용방안

원칙: **무거운 로직(수집·`aptSeq` 매칭·평단가 정규화)은 Java 수집기에, 단순 집계는 Postgres 뷰에.**
뷰어는 서버 없이 Supabase를 직접 읽으므로, Supabase 배터리(Auth·RLS·Storage)가 보안·기능의 핵심이 된다.

**A. DB + 읽기 API (필수, 기본)**
- **Java 수집기 ↔ Supabase**: JDBC(Direct 5432, HikariCP)로 연결해 적재. 마이그레이션은 Flyway로 관리.
- **프론트 ↔ Supabase**: **PostgREST 자동 REST API**로 직접 읽기/쓰기. 별도 조회 서버 불필요.
- 단순 분석(평균 평단가·변동률·신고가)은 **Postgres 뷰/RPC**로 → 프론트는 `SELECT`만. (과설계 회피, SQL이 적합한 영역)

**B. 채택한 배터리 (다중유저·메모·사진으로 채택 확정)**
- **Auth (채택)** — **Google OAuth** 1순위(이메일 매직링크는 보조). 발급 JWT의 `auth.uid()`가 RLS에 직접 물림.
  - ※ **사적 데이터(메모·사진·워치리스트)를 정적 프론트가 직접 다루는 순간 Auth+RLS는 보안 강제 요건** — 유저 1명이어도 필수(public anon 키 노출 구조이므로). 다중유저는 여기에 거의 공짜로 따라옴.
- **RLS (채택, 필수)** — 데이터를 **공용 vs 개인** 두 구역으로 분리(§7 참조).
  - 공용(거래이력·지표뷰): 모두 읽기전용. / 개인(watchlist·notes·photos): `user_id = auth.uid()`로 격리.
- **Storage (채택)** — 유저 **사진 첨부**용 버킷. RLS로 소유자 격리. 무료 1GB(사진 수백 장). (기존 "안 씀" → 쓸 이유 생김.)
- **Realtime (후순위)** — 신고가 하이라이트 실시간 갱신. MVP는 "접속 시 계산"으로 충분. **수집 중→완료 갱신도 Realtime 대신 `collection_job` 폴링**으로(자동 갱신 Realtime은 금도금).

**C. 스케줄러 — GitHub Actions cron**
- 증분 수집은 **GitHub Actions cron이 Java JAR을 매일 실행** → Supabase에 upsert. 무료·언어무관.
- 상주 백엔드가 없으니 "idle-sleep으로 `@Scheduled`가 안 돈다"는 문제 자체가 사라짐(서버가 없으므로).
- (대안: Supabase pg_cron + Edge Function. 하지만 수집 본체가 Java라 Actions가 더 자연스러움.)
- **백필 트리거(자동, 목표):** watchlist 등록/'받기 승인' INSERT → **DB 웹훅 → Edge Function → GitHub `repository_dispatch` → Actions 백필**. **조회(SELECT)는 웹훅이 안 걸리므로 "조회 트리거"는 폐기**(느린 API 재결합 회피). 이 디스패치 체인은 Supabase·GitHub 공식 기능으로 **실현 가능 검증됨**(2026-06-26) — 다중유저 단계의 정식 경로이고, 그전까진 로컬/`workflow_dispatch` 폴백.

**D. 무료 티어 함정 (반드시 인지)**
- Supabase 무료 프로젝트는 **7일간 활동 없으면 자동 일시정지(pause)**.
- 완화: 매일 도는 Actions 증분 수집이 DB에 쓰므로 자연히 활성 유지 → 일시정지 안 걸림(keep-alive 겸).
- DB 백업도 무료 티어는 제한적 → 중요해지면 주기적 `pg_dump`를 별도로.

## 7. 화면 (UI) 및 데이터 구역

### 핵심 3종 (가치의 80%)
1. **워치리스트** — 내 관심단지 카드/리스트. 평당가·최근 변동률·신고가 여부 하이라이트.
2. **비교 테이블** — 선택 단지들을 평당가/기간 변동률로 나란히. 정렬 가능.
3. **변동추이 차트** — 단지별 3/6/12개월 실거래 추세 라인.

### 다중유저·개인기록 (스코프 확장 — 채택)
4. **로그인** — **Google OAuth**(Supabase Auth). 로그인 후 개인 데이터 접근.
5. **단지별 메모** — 유저가 단지에 사적 기록 작성(`user_notes`). "내 앞마당"의 차별점.
6. **사진 첨부** — 메모에 현장 사진 첨부(Supabase Storage).
- 보조: 단지 검색(워치리스트에 담기 위한 것).
- 상태 처리: 로딩 / 비로그인 / 빈 워치리스트 / **미수집(받기 승인 온램프)** / **수집 중(진행 표시)** / 데이터 없음(거래 희소) / **수집 실패(재시도)**.
- **미수집 지역 조회 = '받기 승인' 온램프:** 빈 화면 대신 "이 지역은 아직 안 받아왔어요 — [받기 승인]". 승인 = 백필 트리거이고 **워치리스트 영구등록과 분리**(둘러보기 ↔ 관심 의도 오염 방지). 이후 "수집 중…(수 분, Actions 콜드스타트로 더 걸릴 수 있음)" → 완료 시 채워짐(`collection_job` 폴링). ※ "1~2분" 단정 금지, 정직하게 "수 분".
- **다중유저 경계(MVP):** 새 구 시딩(백필)은 소유자 중심. 임의 유저의 새 지역 자동 백필 전면화는 비범위(자동 디스패치 단계로). 로그인/RLS는 개인데이터 보안 바닥으로 유지.
- **보류(투기 영역):** 유저간 공유·댓글·피드 등 소셜 기능 → 수요 증명 후.

### 데이터 구역 — 공용 vs 개인 (RLS 설계의 축)
```
[공용 — 모든 유저 읽기전용]            [개인 — RLS로 격리, auth.uid() 기준]
· 거래 이력 (수집기 적재)               · watchlist   (내 관심단지, user_id)
· 지표 뷰 (평단가/변동률/신고가)         · user_notes  (내 메모, user_id)
· collection_job (수집 상태·폴링)        · note_photos (Storage 경로, user_id)
  → 공개 정부 데이터라 공유                → 남의 것 못 봄/못 씀
```
무거운 실거래가는 **한 번 수집해 공유**, 유저는 그 위에 **자기 메모·사진만** 얹는다 → 데이터 가볍고 컨셉에 부합.

## 8. 관심단지 등록 — 주소→법정동코드 해석 (지도 활용)

**문제:** 국토부 실거래가 API는 `LAWD_CD`(법정동 시군구 5자리)를 입력으로 받는데, 사용자는 코드를 모른다.
사용자가 아는 건 "단지명" 또는 "주소"뿐. 그 간극을 **지도/지역검색으로 메운다.**

### 등록 흐름
```
사용자: 지도에서 단지 클릭  또는  "강남구 대치동" / 단지명 검색
   ↓
지도·지역 API: 좌표 + 주소 + 법정동코드(10자리) 반환
   ↓
앞 5자리 추출 → LAWD_CD (국토부 API 파라미터)
   ↓
관심단지 레코드에 코드·단지명·법정동 저장 → 이후 수집·필터링에 사용
```

핵심: **지도 API가 주는 법정동코드 앞 5자리 = 국토부 API의 LAWD_CD.**
별도 법정동코드 매핑 테이블을 직접 관리할 필요가 없어진다.

- **수집·백필의 지리 단위는 이 `LAWD_CD`(구) 하나로 고정**(§4.2). 동(읍면동)은 받을 수 없어 저장 후 표시·필터 차원이고, 시 전체는 그 시의 구 `LAWD_CD`들을 루프한다.

### 지도/주소 API 선택지

| 후보 | 강점 | 비고 |
|------|------|------|
| **카카오 지도 API** (추천) | 지도 UI + 키워드(단지명) 검색에서 좌표·법정동코드 한 번에. 무료 쿼터 넉넉(일 30만) | JS 키(지도)·REST 키(검색) 분리 |
| **VWorld** (국토부 공간정보) | 지오코더가 법정동코드 직접 반환. 실거래가와 같은 정부 데이터 생태계 | 지도 UX는 카카오보다 약함 |
| **juso.go.kr** (행안부 도로명주소) | 주소검색 시 `admCd`(법정동코드) 반환 | 지도 없음, 주소검색만 |

→ 지도 UX까지 원하면 **카카오**, 순수 주소→코드 변환만이면 **juso/VWorld**.

### 등록 시 저장 필드
```
Watchlist 단지 = {
  단지명,                 // 응답 필터 키 (필수)
  법정동코드(10자리),      // 지도 API 반환값
  LAWD_CD(5자리),         // 앞 5자리 → 국토부 API 파라미터 (필수)
  법정동(읍면동),          // 동명이 단지 구분 (권장)
  좌표(lat, lng)          // 지도 표시·근처단지 비교용 (선택, UI 안 쓰면 생략)
}
```

### 주의 — 좌표는 선택, 매칭 1차 키는 aptSeq (검증으로 갱신됨)
- **좌표(lat/lng)는 데이터 파이프라인에 불필요.** 지도에 그릴 때만 필요한 UI 전용 값.
- **검증 결과(2026-06-25): 국토부 응답에 단지 식별자 `aptSeq`가 있다.** → "내 단지" 거래 식별의 **1차 키는 `aptSeq`**. 단지명 퍼지매칭(공백·괄호·차수)에 의존하지 않아도 된다.
- 등록 흐름 보정: 지도/검색으로 정규화된 단지명을 얻은 뒤, **해당 단지의 실거래에서 `aptSeq`를 1회 해석해 워치리스트에 저장** → 이후 수집·필터는 `aptSeq`로 정확히 매칭.
  - 한 물리적 단지가 여러 seq로 등록될 수 있어(`현대5차`/`신현대12차`), **단지 1개 ↔ aptSeq N개** 매핑을 허용.
- 단지명·면적은 이제 보조: 등록 시점 aptSeq 해석과 평형 구분(`excluUseAr`)에만 사용. 매칭 순서: **`aptSeq` 1차 → (미상 시) 법정동+단지명 정규화 → 면적**.

## 9. 성공 기준 (Success Criteria)

- 관심단지 N개를 등록하면 3년치 실거래가 자동 백필되어 추이 그래프가 그려진다.
- 단지 2~3개를 평당가 기준으로 한 화면에서 비교할 수 있다.
- 접속 시 신고가/급변 단지가 시각적으로 구분된다.
- 조회는 내 DB에서 나오므로 체감 즉시 (국토부 API 직접 호출 대비 빠름).

## 10. 배포/유통 (Distribution)
- **상주 서버 0개.** 수집기=GitHub Actions(무료), DB+Auth+Storage=Supabase(무료), 프론트=Vercel/Cloudflare Pages(무료).
- 폰 원격 접근: 정적 프론트가 CDN에 항상 떠 있어 즉시 접근(콜드스타트·슬립 없음).
- CI/CD: GitHub Actions가 수집(cron)·프론트 배포 모두 담당. MVP는 수동 배포로 시작 가능.

### 빌드 순서 (스코프는 키우되 한 번에 다 짓지 않기)
```
1. 인사이트 루프   공용 실거래 데이터 → 워치리스트·비교·차트     ← 가치 먼저
2. Auth + RLS      Google 로그인 + 사적 데이터 보안 바닥        ← 메모 전 필수
3. 개인 메모        단지별 내 기록 (user_notes)                ← 차별점
4. 사진 첨부        Storage                                    ← 현장감
   └ 다중유저는 2번 시점에 자동으로 됨. 소셜 기능만 계속 보류.
```

## 내가 본 당신의 사고 방식

- "DB 캐싱은 어느 날짜까지?"라고 먼저 물은 건 좋은 신호다. 대부분 기능부터 그리는데, 당신은 데이터 보존 범위를 먼저 의심했다. 변동추이 기능의 생사가 거기 달려 있었다.

- 무료 배포(Vercel)를 꺼냈을 때 바로 "상시 Java 서버는 못 올라간다"는 제약과 부딪혔는데, 그걸 회피하지 않고 토폴로지 트레이드오프로 끌고 간 게 실용적이다.
